`v7` adds **Playbooks**: a visual DAG editor for orchestrating multiple `workflow_dispatch` calls in sequence, in parallel, or behind manual approval gates. Playbooks are stored in Supabase, runnable from the UI, and exportable as YAML to your repo via the existing PR flow.

## Opening the editor

Click **Playbooks** in the TopBar (between Docs and the theme toggle), or navigate to `#/playbooks`.

The page has a sidebar of playbooks (filtered to the current repo from the [repo switcher](/multi-repo/)) and a [ReactFlow](https://reactflow.dev) canvas on the right.

## Node types

<CardGrid>
  <Card title="Start" icon="rocket">Single entry node. Auto-added when creating a new playbook.</Card>
  <Card title="Dispatch" icon="setting">Triggers one `workflow_dispatch`. Configures workflow file, branch, and inputs.</Card>
  <Card title="Condition" icon="approve-check">Gate based on incoming edges: `all_success`, `any_success`, `always`, or `on_failure`.</Card>
  <Card title="Wait approval" icon="pause">Halts the run until someone clicks **Approve** in `PlaybookRunPanel`.</Card>
  <Card title="Parallel group" icon="up-down-arrows">Annotation/grouping node. Children scheduled concurrently by the frontier loop.</Card>
  <Card title="End" icon="document">Single exit node. Reaching it marks the playbook run `succeeded`.</Card>
</CardGrid>

## Validation

Before saving or running, the editor runs these checks:

- Exactly one `start` and one `end` node.
- No cycles in the DAG.
- Every `dispatch` node has a `workflow_filename` chosen.
- Every edge has both source and target.

## Runner semantics

The DAG executor runs **in-process** on the dispatcher server. Algorithm:

1. Kahn-like topological frontier loop.
2. A node becomes ready when all incoming edges' source nodes are in a terminal state satisfying their `condition` (default `all_success`).
3. **Dispatch nodes** call GitHub's `workflow_dispatch` API, then poll `GET /actions/runs?event=workflow_dispatch&branch=...` every 4 s until the run reaches a terminal conclusion.
4. **Condition nodes** evaluate based on parent statuses and propagate.
5. **Wait-approval nodes** persist `awaiting_approval`, the runner halts; resume via `POST /api/playbook-runs/:id/approve/:nodeId`.
6. **Parallel-group nodes** are pass-through annotations.

| Limit                   | Value                                  |
|-------------------------|----------------------------------------|
| Max concurrent runs     | **10** per server                      |
| Total run timeout       | **60 min**                             |
| Per-dispatch wait       | **30 min**                             |
| Polling interval        | **4 s**                                |
| Orphan recovery         | On server start, `running` rows → `cancelled` |

State is persisted to `playbook_runs.state` (jsonb) after every node transition.

## Endpoints

| Method   | Path                                                  | Notes                                |
|----------|-------------------------------------------------------|--------------------------------------|
| `GET`    | `/api/playbooks?repo_full=`                           | List                                 |
| `GET`    | `/api/playbooks/:id`                                  | Get one                              |
| `POST`   | `/api/playbooks`                                      | Create — body `{repo_full, name, description?, dag}` |
| `PUT`    | `/api/playbooks/:id`                                  | Replace (bumps `version`)            |
| `DELETE` | `/api/playbooks/:id`                                  | Delete                               |
| `POST`   | `/api/playbooks/:id/run`                              | Kick off runner — needs `x-github-pat` |
| `POST`   | `/api/playbooks/:id/export-pr`                        | Export YAML to `.github/playbooks/{slug}.yml` via PR flow |
| `GET`    | `/api/playbook-runs/:id`                              | Current state                        |
| `POST`   | `/api/playbook-runs/:id/cancel`                       | Cancel                               |
| `POST`   | `/api/playbook-runs/:id/approve/:nodeId`              | Resume past a `wait_approval` gate   |

## YAML export

**Export PR** writes a `.github/playbooks/{slug}.yml` file to a fresh branch `dispatcher/playbook/{slug}` and opens a PR against the repo's default branch. The YAML preserves the full DAG (nodes + edges) — it's documentary, not GHA-native:

```yaml
# generated by gha-dispatcher
name: ship-frontend
description: build → smoke → deploy-canary → wait-approval → deploy-prod
repo: pv-udpv/pplx-lab
nodes:
  - id: n1
    kind: dispatch
    data: { workflow_filename: build.yml, branch: master, inputs: {} }
  # ...
edges:
  - { source: start, target: n1 }
  # ...
```

## Schema

```sql
CREATE TABLE gha_dispatcher.playbooks (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  repo_full    text NOT NULL,
  name         text NOT NULL,
  description  text,
  dag          jsonb NOT NULL,
  version      int  NOT NULL DEFAULT 1,
  created_at   timestamptz DEFAULT now(),
  updated_at   timestamptz DEFAULT now(),
  UNIQUE (repo_full, name)
);

CREATE TABLE gha_dispatcher.playbook_runs (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  playbook_id  uuid NOT NULL REFERENCES gha_dispatcher.playbooks(id) ON DELETE CASCADE,
  status       text NOT NULL DEFAULT 'pending',
  state        jsonb NOT NULL DEFAULT '{}',
  started_at   timestamptz DEFAULT now(),
  completed_at timestamptz,
  triggered_by text
);
```

Both tables have RLS enabled with a `service_role`-only policy.
